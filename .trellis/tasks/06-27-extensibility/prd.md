# Custom tools and extensibility (allowlist, managers, todo/plan)

## Goal

Let a power user's own pi extensions, custom tools, and skills actually work in the desktop app — auto-discovered tools load AND activate (not silently dropped), load errors are visible, skills can be browsed/authored, and the agent's own plan/todo list is rendered — while keeping mutation safety (custom tools gated by default) and adding a true plan mode that narrows the toolset.

## Background

The desktop is a pi-citizen harness, but today it pins the toolset to a fixed 7-tool array and never reads back anything pi discovered from `~/.pi/agent/extensions`, so a user's own extension tools are invisible and inert:

- `SessionController.buildSession` calls `createAgentSession({ ..., tools: TOOLS })` with `const TOOLS = ["read","grep","find","ls","bash","edit","write"]` (`apps/desktop/src/main/agent/sessionController.ts:32`, used at `:159`). In the SDK, `allowedToolNames = options.tools ?? ...` (`packages/coding-agent/src/core/sdk.ts:245`) — so passing `tools` turns it into a hard allowlist. `_refreshToolRegistry` then filters every tool through `isAllowedTool` (`packages/coding-agent/src/core/agent-session.ts:2293-2303`), dropping any extension/custom tool not in the 7-array. The user's auto-discovered extension tools are loaded into the registry and then removed.
- The SDK already exposes the right primitives the desktop ignores: `getAllTools()` (`agent-session.ts:781`), `getActiveToolNames()` (`:774`), `setActiveToolsByName()` (`:801`), and `initialActiveToolNames` / `allowedToolNames` / `excludedToolNames` config (`:171-175`). Omitting `tools` keeps built-ins + extension tools; the catch is the SDK's default *active* set is only `["read","bash","edit","write"]` (`sdk.ts:244`), so the desktop must re-assert `grep/find/ls` (and any extension tools) as active rather than relying on `tools` to do it.
- Extension load failures are captured but surfaced nowhere. `DefaultResourceLoader.getExtensions()` returns `{ extensions, errors, runtime }` with `errors: Array<{ path; error }>` (`packages/coding-agent/src/core/resource-loader.ts:33,257`; loader at `packages/coding-agent/src/core/extensions/loader.ts:420`). The desktop builds a resource loader and `await resourceLoader.reload()` (`sessionController.ts:145-151`) but never reads `.getExtensions().errors`. The pi CLI does surface them (`packages/coding-agent/src/main.ts:656,725`). There is no IPC channel for it in `apps/desktop/src/shared/ipc.ts`.
- Approvals only gate three hardcoded built-ins. `const MUTATING = new Set(["bash","edit","write"])` (`apps/desktop/src/main/agent/approval.ts:4`); the gate returns `undefined` (auto-allow) for everything else (`approval.ts:16`). So once custom tools load, an unknown custom tool would auto-run with no prompt — there is no policy for them.
- Tool-result `details` already cross IPC as an open passthrough: `IpcToolResult.details?: { ...; [k: string]: unknown }` (`apps/desktop/src/shared/ipc.ts:133-138`), populated verbatim from the SDK result in `mappers.ts:137-144`. So a plan/todo tool's `details.todos` already reaches the renderer with zero new SDK or IPC plumbing — it is simply never rendered (no component reads it; see `apps/desktop/src/renderer/src/components/` — `ToolChip.tsx`, `BashCard.tsx`, etc., none handle todos).
- Skills already feed the slash menu but cannot be created or inspected. `loadSkills(...)` is used in `SessionController.listCommands` (`sessionController.ts:395-403`) to populate `/skill:` commands; `loadSkills` is a public export (`packages/coding-agent/src/index.ts:235`) and validation lives in the SDK (`packages/coding-agent/src/core/skills.ts` — `validateName` `:92`, `validateDescription` `:117`, `Skill`/`LoadSkillsResult` shapes `:74-86`). There is no UI to list, validate, or author a SKILL.md.
- Permission mode is global and only denies at approval time. `readonly` makes `requestApproval` return `false` (`sessionController.ts:113`); the model still *sees* edit/write/bash in its toolset and tries them, then gets blocked. There is no mode that narrows the active toolset or injects a planning prompt (`PermissionMode = "ask"|"acceptEdits"|"yolo"|"readonly"` in `ipc.ts:59`; pool-global `setMode` at `sessionPool.ts:329`, read via `getMode` at `:87`).

## Requirements

- **(AGENT-1) Stop the fixed-7 allowlist from disabling user tools; gate unknown tools by policy.** Build sessions so extension/custom tools auto-discovered from `~/.pi/agent/extensions` (and project `.pi/extensions`) actually load AND are active, while the desktop's intended built-in set (`read,grep,find,ls,bash,edit,write`) stays active. Treat any tool the desktop doesn't recognize as built-in as "custom" and gate it through the approval flow by default (default-deny / prompt), so a newly loaded custom tool can never auto-run silently. User value: a power user's own tools just work in the app; dev value: unblocks AGENT-2/AGENT-3. Evidence: `sessionController.ts:32,159`; `sdk.ts:245` (`allowedToolNames = options.tools`); `agent-session.ts:2293-2303,2360-2378` (allowlist filtering); `agent-session.ts:774,781,801` (`getActiveToolNames`/`getAllTools`/`setActiveToolsByName`); `approval.ts:4,16` (`MUTATING` set). sdkSupport: yes — omit `tools` (keep built-ins + extension tools) and use `initialActiveToolNames` to assert `grep/find/ls`, OR `getAllTools()` + `setActiveToolsByName()` post-build; gate non-built-in tool names in `approval.ts`. Effort: M.

- **(AGENT-2) Extensions / custom-tools manager with load-error diagnostics.** A read-only manager surface (new IPC + DTO + a Settings section) that lists the loaded extensions and their registered custom tools, and prominently shows extension load errors (path + message) that are currently swallowed. User value: when an extension fails to load the user sees *why* instead of a silently missing tool. Evidence: `resource-loader.ts:33,257` and `extensions/loader.ts:420` (`errors: Array<{ path; error }>`); desktop `await resourceLoader.reload()` at `sessionController.ts:145-151` never reads `.getExtensions().errors`; CLI precedent `main.ts:656,725`; no channel in `ipc.ts`. Tool inventory from `session.getAllTools()` (`agent-session.ts:781`, returns name/description/sourceInfo). sdkSupport: yes (`getExtensions()`, `getAllTools()` with `sourceInfo.source` distinguishing `builtin`/`sdk`/extension). Effort: M.

- **(AGENT-3) Plan/todo panel rendered from tool-result `details.todos`.** When a tool result carries a `details.todos` array (a plan/todo-writing tool), render it as a live checklist panel in the transcript (styled per DESIGN.md). Zero new SDK work and zero new IPC: `details` is already an open passthrough across the bridge. User value: the agent's plan is legible at a glance instead of buried in a tool card. Evidence: `ipc.ts:133-138` (`details?: { ...; [k:string]: unknown }`); `mappers.ts:137-144` (passes `result.details` verbatim) and `mappers.ts:181-190` (transcript hydration keeps `details`); no renderer component reads todos today. sdkSupport: n/a for transport (already crosses); renderer-only feature consuming the existing DTO. Effort: S–M.

- **(AGENT-4) Skills authoring / browser UI.** A Settings/manager view that lists discovered skills (name, description, source/scope) and lets the user create or edit a SKILL.md, validating name + description against the SDK's rules before writing to `~/.pi/agent/skills/<name>/SKILL.md`. User value: skills become first-class and authorable from the app (then immediately appear in the `/skill:` menu). Evidence: `loadSkills` already used at `sessionController.ts:395-403`; public export `index.ts:235`; validation + types in `skills.ts:74-86,92,117`. sdkSupport: partial — `loadSkills`/validators exist; the *write* (mkdir + SKILL.md with frontmatter) is new desktop main-process file I/O (mirrors how `models.json` is written in `sessionPool.ts:477`). New IPC for list/validate/write. Effort: M–L.

- **(AGENT-10) Plan-mode toggle that narrows the active toolset + injects a planning prompt.** A new mode (distinct from `readonly`) that, while active, restricts the *active* tool set to read-only tools (`read,grep,find,ls` + any read-only custom tools) via `setActiveToolsByName` so the model never even attempts a mutation, and appends a planning system-prompt nudge so it produces a plan instead of acting. Unlike `readonly` (which only denies at the approval gate after the model already tried), this changes what the model sees. User value: a genuine "plan first" mode. Evidence: `readonly` denies post-hoc at `sessionController.ts:113`; `setActiveToolsByName` (`agent-session.ts:801`) + `getAllTools` (`:781`) enable narrowing; system-prompt append precedent = `IMAGE_OUTPUT_INSTRUCTION` via `appendSystemPrompt` (`sessionController.ts:47-51,149`); mode enum `ipc.ts:59`; pool-global mode `sessionPool.ts:87,329`. sdkSupport: yes. Effort: M. (Depends on AGENT-1's tool-activation handling.)

## Acceptance Criteria

- [ ] (AGENT-1) With a user extension that registers a custom tool present under `~/.pi/agent/extensions`, the tool appears in the live session's tools (verified via the AGENT-2 manager / `getAllTools()`), and the built-in `read,grep,find,ls,bash,edit,write` stay active; removing the `tools: TOOLS` allowlist does not drop `grep/find/ls`.
- [ ] (AGENT-1) A non-built-in (custom/unknown) tool is gated by default: it triggers the approval dialog (or is denied) rather than auto-running; built-in read-only tools still auto-run and `bash/edit/write` still gate as before.
- [ ] (AGENT-2) A new IPC method returns the loaded extensions, their custom tools, and any load errors (path + message); a Settings section renders them, and an intentionally broken extension shows its error text in the UI.
- [ ] (AGENT-3) A tool result carrying `details.todos` renders as a checklist panel in the transcript (live during a turn and after reload via `mapTranscript`), styled per DESIGN.md; results without todos are unaffected.
- [ ] (AGENT-4) The skills view lists discovered skills with source/scope; creating a skill validates name/description against the SDK rules, writes `~/.pi/agent/skills/<name>/SKILL.md`, and the new skill subsequently appears in the composer `/skill:` menu (`listCommands`).
- [ ] (AGENT-10) Activating plan mode narrows the active toolset to read-only tools (model cannot call edit/write/bash) and the response is plan-oriented; leaving plan mode restores the full active set. Plan mode is visibly distinct from `readonly` in the UI.
- [ ] `npm run typecheck && npm run lint && npm run test && npm run build` all green (from `apps/desktop`).
- [ ] The renderer stays pi-free: no `@earendil-works/pi-*` import in `src/renderer/**`; every new capability is a channel + DTO in `src/shared/ipc.ts`, wired in preload + `bridge.ts`, mapped in main.
- [ ] Token-free verification where possible: light + dark screenshots (`PI_SHOT` / `PI_THEME=dark`) of the new manager/skills/todo panel; AGENT-1/AGENT-10 tool-activation covered by a main-process unit test (no model call).

## Design hints (for the later design.md)

- **AGENT-1 (core, do first):** in `SessionController.buildSession` drop `tools: TOOLS` from `createAgentSession`. Keep built-ins + extension tools by omitting `tools`; then assert the desired active set with `initialActiveToolNames: ["read","grep","find","ls","bash","edit","write"]` (SDK config `agent-session.ts:171`) OR post-build `session.setActiveToolsByName([...getActiveToolNames-derived, "grep","find","ls"])`. For default-deny of customs: in `approval.ts`, instead of `MUTATING.has(name)`, treat as "requires approval" any tool NOT in a known auto-run read-only allowlist (`read,grep,find,ls`) — i.e. invert to allowlist-of-auto-run so unknown customs gate. Decide the exact policy (gate-by-default vs a per-tool toggle) in design.
- **AGENT-2:** add `listExtensions()` IPC → DTO `{ extensions: {path; tools: string[]}[]; tools: {name; description; source}[]; errors: {path; message}[] }`. Source it in the controller/pool from `resourceLoader.getExtensions()` (errors) + `session.getAllTools()` (`sourceInfo.source`). Wire in `bridge.ts` + `preload/index.ts`; render in `SettingsPanel.tsx`.
- **AGENT-3:** renderer-only. Add a `TodoPanel.tsx` (or branch inside the tool-card render) that reads `toolResult.details.todos`; wire it where tool results render (alongside `BashCard`/`DiffView` in `MessageList.tsx`); style in `styles/tools.css` per DESIGN.md. No `ipc.ts` change.
- **AGENT-4:** new IPC `listSkills()` / `validateSkill()` / `writeSkill()`; main uses `loadSkills` (already imported in `sessionController.ts`) + the SDK validators from `skills.ts`, and writes the SKILL.md via `node:fs` (mirror the `models.json` write in `sessionPool.ts`). UI in `SettingsPanel.tsx` (or a new skills view). After write, re-run `listCommands` so `/skill:` updates.
- **AGENT-10:** extend `PermissionMode` in `ipc.ts` with `"plan"`, or add a separate boolean if it must compose with the existing 4 modes (decide in design). On enter, `session.setActiveToolsByName(readOnlySet)` and append a planning instruction (same `appendSystemPrompt` path as `IMAGE_OUTPUT_INSTRUCTION`, or re-derive on rebuild); on exit restore. Mode is pool-global today (`sessionPool.ts:329`) — confirm whether plan mode is per-session or global.
- Keep `runExclusive` serialization for any new lifecycle-touching op (`setActiveToolsByName` on rebuild) per the repo's serialize rule.

## Dependencies / sequencing

- **AGENT-1 first** — it unblocks AGENT-2 (tool inventory), AGENT-3 (a custom todo tool only loads once customs aren't dropped), and AGENT-10 (toolset narrowing reuses the same activation path).
- AGENT-3 is independent of AGENT-1 *if* the todos come from a built-in/existing source, but in practice the todo/plan tool is a custom/extension tool, so it benefits from AGENT-1 landing first.
- AGENT-2 and AGENT-4 are independent of each other (both are Settings manager surfaces).
- Sibling children of `06-27-desktop-roadmap`: none hard-blocking; coordinate only on `ipc.ts` channel additions to avoid merge churn.

## Out of scope

- A general extension *installer* / marketplace, enabling-disabling extensions from the UI, or editing extension source.
- Authoring/editing prompt templates or themes (skills only for AGENT-4).
- Per-tool granular permission persistence beyond the session (the existing `sessionAllow` "always" semantics are reused, not redesigned).
- Designing a new todo/plan *tool*; AGENT-3 only renders whatever a tool emits in `details.todos`.
- MCP servers / remote tools.
